PPO Agent — Environment & Training Setup

Gymnasium env wraps the frozen LSTM · Discrete(4) actions · Stable-Baselines3 PPO · ~500k timesteps

What is the PPO agent doing here?

The LSTM learns building physics from data. The PPO agent learns control strategy by interacting with a simulated version of that physics. Instead of real sensors, it uses the LSTM as a fast, cheap simulator — it can run millions of virtual hours overnight and never damage real equipment.

The frozen LSTM is the physics engine. PPO is the player learning to control it.

Gymnasium environment — what PPO sees and does

Observation space — what the agent sees

Box(shape=(26,), dtype=float32)
24 values: last 24h T_inside (norm)
1 value: current T_outside (norm)
1 value: current solar radiation (norm)

Why T_inside history? Concrete floor has 8-24h thermal memory — the last few hours tell the agent whether the building is warming or cooling.

Action space — what the agent controls

Discrete(4)
0 → fan=0, heater=0 (both off)
1 → fan=1, heater=0 (fan only)
2 → fan=0, heater=1 (heater only)
3 → fan=1, heater=1 (both on)

Four mutually-exclusive combos map cleanly to one discrete integer. SB3 PPO handles this natively.

Episode structure — how one training episode works

reset() Sample random start hour from training data → load last 24h real data as initial window
↓ start at a random point in the year so PPO sees all seasons
step(action)
1. Decode action → fan_on, heater_on
2. Build LSTM input: [T_outside, T_inside, T_floor, SR, fan_on, heater_on] (normalised)
3. Call frozen LSTM → predict [T_inside_next, T_floor_next]
4. Slide 24h window forward (drop oldest row, append new)
5. Compute reward(T_inside_next, fan_on, heater_on)
6. Return obs, reward, done=False (episode runs fixed length)
done after 168 steps (= 7 simulated days per episode)

168 steps (7 days) is long enough to see seasonal patterns, short enough that many episodes fit in 500k timesteps. At 168 steps/episode → ~2976 episodes.

What gets saved

(zip unpacks to: policy.pth, replay_buffer, hyperparams)
At runtime: model = PPO.load("models/ppo_hvac.zip") → action, _ = model.predict(obs)

Why the LSTM is frozen during PPO training

If the LSTM weren't frozen: PPO's gradient updates would flow through the LSTM and distort the physics model — the building would gradually "learn" to be easy to control rather than accurately modelling the real building.

So: After Phase 1, we call lstm.eval() and torch.no_grad() inside the env's step(). The LSTM becomes a fixed physics oracle. Only PPO's own policy weights get updated.
