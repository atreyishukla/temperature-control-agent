Which architecture approach?

Fan on/off · Heater on/off · Online learning · Edmonton building

D

Train in Python → Export model.json + weights → Node-RED loads files (Recommended)

Train LSTM and PPO in Python (TF/Keras for LSTM, Stable-Baselines3/PyTorch for PPO policy). Export each to TFJS format: lstm/model.json + lstm.weights.bin, policy/model.json + policy.weights.bin, scaler.json (normalization params). The Node-RED custom node loads all three at startup — no Python process needed at runtime. For continual learning: Node-RED logs real experience to a file; a background Python script re-trains periodically and overwrites the exported files; Node-RED hot-reloads the new weights.

Pros

  • No Python process running alongside Node-RED
  • Self-contained Node-RED node — just drop in the files
  • Best of both worlds: Python training ergonomics + JS runtime
  • MPC = random shooting in TFJS (works perfectly with discrete fan/heater)
  • Online learning via scheduled Python retrain → hot-reload in Node-RED
  • Each exported artifact is inspectable and replaceable independently

Cons

  • Learning happens in batches (e.g. every 24h), not step-by-step
  • PyTorch PPO policy → ONNX → TFJS conversion adds one extra export step
A

PyTorch throughout + Python HTTP server

LSTM + PPO in PyTorch. A small Flask/FastAPI server stays alive. Node-RED sends sensor data → Python runs MPC + PPO → returns fan/heater command. Python process does online learning step-by-step from each real observation.

Pros

  • True step-by-step online learning
  • SB3 PPO is the easiest RL library available

Cons

  • Python process must stay running — if it crashes, Node-RED goes blind
  • More moving parts to manage in production
C

TF/Keras throughout + TFJS (pure JS in Node-RED)

Train everything in TF/Keras. Export to TFJS. Node-RED does inference AND fine-tuning with model.fit() in JavaScript — no Python ever at runtime.

Pros

  • Zero Python at runtime
  • True in-node online fine-tuning possible via TFJS

Cons

  • TFJS online training is experimental, memory-hungry
  • PPO in TF-Agents is much harder than SB3
  • Harder to debug — no Python tooling at training time

How "keep learning" works in Option D

Node-RED appends each real timestep to experience.csv (state, fan_action, heater_action, T_inside_actual). A Python script runs on a schedule (e.g. every 24h or every N hours of new data): reads experience.csv → fine-tunes LSTM on new observations → fine-tunes PPO on real rewards → overwrites model.json + *.bin files → Node-RED detects file change and reloads weights. Models improve continuously without a persistent Python process.
