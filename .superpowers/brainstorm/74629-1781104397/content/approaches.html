Which architecture approach?

Fan on/off + Heater on/off outputs · Edmonton building · Online learning required

A

PyTorch throughout + Python HTTP server (Recommended)

LSTM and PPO both trained in PyTorch. Stable-Baselines3 handles PPO — best-in-class RL library. MPC uses random shooting (ideal for discrete fan/heater actions). A lightweight Python HTTP server sits between Node-RED and the models. Node-RED sends sensor data → Python does MPC+PPO inference+learning → returns commands.

Pros

  • SB3 PPO is the easiest, most documented PPO available
  • Online learning with SB3 is natural (model.learn() loop)
  • MPC via random shooting is simpler and more reliable with discrete actions than gradient-based Adam
  • One framework, one mental model, clean separation
  • Python process stays alive → continual learning just works

Cons

  • Python process must run continuously alongside Node-RED
  • Slightly more setup than a pure TFJS node
B

PyTorch throughout + subprocess (no HTTP)

Same PyTorch stack, but Node-RED spawns a Python child process per request instead of keeping a server alive. Simpler to wire up initially, but each call pays process startup cost and you lose the persistent learning state between calls.

Pros

  • No persistent server to manage
  • Dead simple Node-RED integration (exec node)

Cons

  • Model weights reload from disk every hour — slow
  • Online learning state lost between calls
  • Defeats the purpose of "keep running and learning"
C

TensorFlow/Keras + TensorFlow.js (pure JS in Node-RED)

Train LSTM and PPO policy in TF/Keras. Export to TFJS SavedModel format. The Node-RED custom node loads both models in JavaScript, runs MPC with Adam gradients natively through TFJS, and does online weight updates in JS.

Pros

  • No Python process at runtime — everything inside Node-RED
  • Native TFJS gradients for MPC (Adam works on continuous relaxation)

Cons

  • TF-Agents PPO is far less ergonomic than SB3
  • TFJS online training is experimental and poorly documented
  • Discrete action PPO in pure JS is painful to implement
  • Harder to debug and iterate on

Key insight for discrete actions

The original spec described Adam-based MPC — this works well for continuous kW values. With binary fan/heater on/off, gradient-based optimization is problematic (gradients vanish at the 0/1 boundary). Random shooting (sample ~1000 random 24h on/off sequences, simulate each with LSTM, pick best) is the right tool for discrete MPC — faster, simpler, and more reliable.
