MPC — Random Shooting

Thinks 24 hours ahead · Samples 1024 candidate schedules · Picks the best first action

Why MPC exists alongside PPO

PPO is reactive — it sees the current state and picks an action. It has no explicit lookahead; it trusts that its learned value function captures future consequences.
MPC is a planner — every hour it simulates 1024 possible 24-hour futures and picks the schedule that scores best. It's slower but it can see consequences that PPO's shallow MLP might miss: e.g. "if I run the heater now, the floor will still be warm in 6 hours and I won't need it then."
At runtime we blend both: 70% MPC action + 30% PPO action.

Why random shooting (not gradient-based MPC)?

Gradient-based MPC (e.g. Adam on action parameters) requires differentiable actions. Our actions are binary integers — the gradient of "fan=0 vs fan=1" is zero everywhere and undefined at the boundary. Gradient descent can't cross it.

Random shooting sidesteps this entirely: just sample random binary schedules, simulate each one with the LSTM, score by total reward, pick the best. No gradients needed. Works perfectly for discrete on/off control.

The algorithm — step by step

Input current_window: last 24h real sensor data, shape (24, 6)
Input weather_forecast: T_outside + SR for next 24h, shape (24, 2)
Step 1 Sample 1024 random schedules
schedules = np.random.randint(0, 2, size=(1024, 24, 2))
→ shape (1024, 24, 2): 1024 candidates, 24 hours, [fan, heater]
Step 2 Simulate each schedule autoregressively (LSTM rollout)
for each candidate k in 0..1023:
window_k = current_window.copy()
total_reward[k] = 0
for t in 0..23:
T_next = lstm(window_k) # [T_inside, T_floor]
r = reward(T_next[0], schedules[k,t])
total_reward[k] += gamma^t * r # discounted sum
window_k = slide(window_k, T_next, weather[t], schedules[k,t])
Step 3 Pick the winning schedule
best_k = argmax(total_reward)
mpc_action = schedules[best_k, 0] ← only the FIRST step matters
Output mpc_action: [fan_on, heater_on] for this hour

⚠ Receding horizon: we only execute the first step of the winning schedule. Next hour, we re-run MPC with fresh sensor data. The 24-step plan is a scoring tool, not a commitment.

Performance on CPU

Batching trick — vectorise across 1024

Instead of looping over 1024 candidates one by one,
run the LSTM on all 1024 simultaneously:
input shape: (1024, 24, 6)
output shape: (1024, 2)
PyTorch broadcasts this in one forward pass.

Timing

1024 × 24-step rollout (batched): ~80-150ms on CPU
We have a full hour between decisions.
Even 10× slower is fine.

No GPU required.

Blending MPC + PPO at runtime

# Both return a [fan_on, heater_on] pair
mpc_action = mpc.solve(window, weather_forecast) # [0/1, 0/1]
ppo_action = ppo.predict(obs) # integer 0-3 → decode
# Blend: 70/30 vote on each device independently
fan = 1 if (0.7*mpc_action[0] + 0.3*ppo_action[0]) >= 0.5 else 0
heater = 1 if (0.7*mpc_action[1] + 0.3*ppo_action[1]) >= 0.5 else 0
# Result: final discrete fan/heater command sent to actuators

The 70/30 split means MPC dominates but PPO can override when it's confident — e.g. if MPC says fan=off but PPO strongly predicts fan=on (0.7×0 + 0.3×1 = 0.3 < 0.5, so MPC wins). If both agree, the result is unanimous. You can tune this ratio after testing.
