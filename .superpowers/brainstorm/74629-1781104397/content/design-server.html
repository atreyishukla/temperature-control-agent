Flask Server + Online Learning Loop

HTTP bridge for Node-RED · experience.csv logging · periodic retrain

Flask server — the runtime interface

# Two endpoints
POST /predict
Body: { T_outside, T_inside, T_floor, SR_direct, timestamp }
Server maintains a rolling 24h window in memory
Returns: { fan_on: 0|1, heater_on: 0|1, source: "mpc"|"ppo"|"blend" }
POST /log
Body: { ...same fields + actual T_inside_next (measured next hour) }
Appends one row to experience.csv
Returns: { ok: true, rows_logged: N }

Node-RED calls /predict every hour with fresh sensor data, then calls /log the next hour once the actual outcome is known.

experience.csv — what gets logged

timestamp, T_outside, T_inside, T_floor, SR_direct, fan_on, heater_on, T_inside_actual
2026-06-10T14:00, -5.2, 19.1, 17.3, 0.0, 0, 1, 20.4
2026-06-10T15:00, -4.8, 20.4, 18.1, 12.3, 0, 0, 20.1
...

T_inside_actual is what the sensor read one hour after the action — the ground truth the LSTM was trying to predict. This is the supervised signal for LSTM retraining.

Online learning loop

1

Every hour: log real experience

Node-RED sends actual outcome → /log appends row to experience.csv. No computation, just bookkeeping.

2

Every 24h (or every 168 rows): retrain LSTM

Python script reads experience.csv, fine-tunes lstm_best.pt on new rows (10 epochs, small lr=1e-4), saves updated weights. Server hot-reloads: lstm.load_state_dict(torch.load(...)).

3

Every 7 days: retrain PPO

PPO retrains inside the updated LSTM simulator using the same Gymnasium env. Replaces ppo_hvac.zip. Server hot-reloads the new policy.

⚠ LSTM retrains first (it's the simulator PPO learns in). PPO always retrains on the latest LSTM. Order matters.
